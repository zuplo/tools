# Zuplo Project Setup Guide

Step-by-step guide for setting up a new Zuplo API gateway project. Walks developers through the most common initial tasks from project creation through production deployment.

## Critical rule: Read docs before configuring

Before configuring ANY Zuplo feature, you MUST read the relevant documentation first. Do not rely on training data. Use these sources in priority order:

1. **Local docs (preferred):** Read from `node_modules/zuplo/docs/` — version-matched, always available offline. Check both the project root and parent directories for monorepos.
2. **MCP server tools:** Use `search-zuplo-docs` and `ask-question-about-zuplo` if the Zuplo MCP server is connected.
3. **Fetch docs via URL:** Fetch from `https://zuplo.com/docs/llms.txt` for a page index, then fetch individual pages.

## Step 1: Create the project

```bash
npx create-zuplo-api@latest my-api
cd my-api
npm install
npm run dev
```

This starts the local gateway (API on `:9000`, route designer on `:9100`, docs on `:9200`).

Use `--empty` for a blank project or `--example <name>` for a template (e.g. `--example basic-api-gateway`). Browse templates at `https://zuplo.com/examples`.

### Project structure

```
my-api/
├── config/
│   ├── routes.oas.json     # Route definitions (OpenAPI 3.1 + x-zuplo-route)
│   └── policies.json       # Named policy instances
├── modules/                # Custom TypeScript handlers and policies
├── docs/                   # Developer portal (Zudoku-based)
├── tests/                  # API tests
├── zuplo.jsonc             # Project config
├── package.json
├── .env                    # Local env vars (do NOT commit)
└── .env.zuplo              # Generated by `zuplo link` (do NOT commit)
```

## Step 2: Define routes and proxy to a backend

Routes are defined in `config/routes.oas.json` as an OpenAPI 3.1 spec. Each route uses the `x-zuplo-route` extension to attach a handler and policies.

### Minimal route proxying to a backend

```json
{
  "paths": {
    "/v1/todos": {
      "x-zuplo-path": {
        "pathMode": "open-api"
      },
      "get": {
        "summary": "List todos",
        "operationId": "listTodos",
        "x-zuplo-route": {
          "corsPolicy": "none",
          "handler": {
            "export": "urlForwardHandler",
            "module": "$import(@zuplo/runtime)",
            "options": {
              "baseUrl": "https://your-backend.example.com",
              "forwardSearch": true
            }
          },
          "policies": {
            "inbound": [],
            "outbound": []
          }
        }
      }
    }
  }
}
```

**Key handler options:**

- `baseUrl` — The upstream URL to forward to. The request path is appended.
- `forwardSearch` — Pass query parameters through to the backend.

You can also use the local Route Designer at `http://localhost:9100` to visually create and edit routes.

For full route configuration details: [`references/setup-steps.md`](references/setup-steps.md)

## Step 3: Add rate limiting

Rate limiting is one of the most important policies to add early. Add it as an inbound policy on your routes.

### 1. Define the policy in `config/policies.json`

```json
{
  "name": "rate-limit-inbound",
  "policyType": "rate-limit-inbound",
  "handler": {
    "export": "RateLimitInboundPolicy",
    "module": "$import(@zuplo/runtime)",
    "options": {
      "rateLimitBy": "ip",
      "requestsAllowed": 100,
      "timeWindowMinutes": 1
    }
  }
}
```

### 2. Attach to routes in `config/routes.oas.json`

```json
"policies": {
  "inbound": ["rate-limit-inbound"]
}
```

Rate limiting options include `ip`, `user`, and `function` for the `rateLimitBy` field. For production, use `user` after adding authentication so limits apply per consumer.

**Docs:** Fetch `https://zuplo.com/docs/policies/rate-limit-inbound` for full options.

## Step 4: Add authentication

The most common auth methods for new projects are API key authentication and JWT authentication. You can use one or both.

### API key authentication

```json
{
  "name": "api-key-inbound",
  "policyType": "api-key-inbound",
  "handler": {
    "export": "ApiKeyInboundPolicy",
    "module": "$import(@zuplo/runtime)",
    "options": {}
  }
}
```

Attach to routes: `"inbound": ["api-key-inbound", "rate-limit-inbound"]`

After adding the policy, set up the API Key Service:
1. In the Zuplo Portal, go to **Services > API Key Service**
2. Click **Configure** to create an API key bucket
3. Create a consumer and generate an API key
4. Clients send the key in the `Authorization: Bearer <key>` header

### JWT authentication (OpenID Connect)

For JWT-based auth (Auth0, Cognito, Firebase, etc.), use the OpenID JWT policy. Fetch the doc page for your specific provider:

- `https://zuplo.com/docs/policies/open-id-jwt-auth-inbound`
- `https://zuplo.com/docs/policies/auth0-jwt-auth-inbound`
- `https://zuplo.com/docs/policies/cognito-jwt-auth-inbound`

**Docs:** Fetch `https://zuplo.com/docs/policies/api-key-inbound` for full API key options.

For detailed auth setup: [`references/setup-steps.md`](references/setup-steps.md)

## Step 5: Configure CORS

Every route has a `corsPolicy` field in `x-zuplo-route`. Built-in options:

- `none` — No CORS headers (default)
- `anything-goes` — Allows any origin, method, header (not for production)

### Custom CORS policy

Define in `config/policies.json` under the `corsPolicies` array:

```json
{
  "corsPolicies": [
    {
      "name": "my-cors-policy",
      "allowedOrigins": [
        "https://app.example.com",
        "https://admin.example.com"
      ],
      "allowedMethods": ["GET", "POST", "PUT", "DELETE"],
      "allowedHeaders": ["Authorization", "Content-Type"],
      "exposeHeaders": ["X-Request-Id"],
      "maxAge": 3600,
      "allowCredentials": true
    }
  ]
}
```

Then reference it on routes:

```json
"x-zuplo-route": {
  "corsPolicy": "my-cors-policy"
}
```

Supports wildcard subdomains (e.g. `https://*.example.com`) and environment variables in origins.

**Docs:** Fetch `https://zuplo.com/docs/articles/cors` for full CORS guide.

## Step 6: Set environment variables

Environment variables store configuration and secrets outside source code.

### Local development

Create a `.env` file in the project root (not committed to git):

```
BACKEND_URL=https://api.example.com
BACKEND_SECRET=my-secret-value
```

### Remote environments

Set via the Zuplo Portal (**Settings > Environment Variables**) or CLI:

```bash
zuplo variable create --name BACKEND_SECRET --value "my-secret" --is-secret true --branch main
```

### Using in code

```ts
import { environment } from "@zuplo/runtime";

const backendUrl = environment.BACKEND_URL;
```

### Using in configuration

Use the `$env()` selector in policy and route configuration:

```json
{
  "options": {
    "baseUrl": "$env(BACKEND_URL)"
  }
}
```

Variables apply per environment (prod, preview, development). Changes require redeployment.

**Docs:** Fetch `https://zuplo.com/docs/articles/environment-variables` for full guide.

## Step 7: Secure the backend

Ensure your backend only accepts traffic through the gateway. The simplest approach is a shared secret header.

### 1. Create an environment variable `BACKEND_SECRET`

### 2. Add a Set Headers policy

```json
{
  "name": "set-backend-secret",
  "policyType": "set-headers-inbound",
  "handler": {
    "export": "SetHeadersInboundPolicy",
    "module": "$import(@zuplo/runtime)",
    "options": {
      "headers": [
        {
          "name": "x-backend-secret",
          "value": "$env(BACKEND_SECRET)"
        }
      ]
    }
  }
}
```

### 3. Verify the header on your backend

Check for the `x-backend-secret` header on incoming requests and reject any that don't match.

Other backend security options: mTLS, cloud provider IAM (GCP, AWS, Azure), secure tunnels. See `https://zuplo.com/docs/articles/securing-your-backend`.

## Step 8: Set up the developer portal

Zuplo includes a built-in developer portal powered by Zudoku. It auto-generates API documentation from your OpenAPI spec.

The portal lives in the `docs/` directory and is configured via `docs/zudoku.config.ts`.

Start the docs server standalone:

```bash
zuplo docs
# or with a custom port
zuplo docs --port 3000
```

The portal automatically includes:
- API reference generated from `routes.oas.json`
- Interactive API explorer for testing endpoints
- Markdown documentation pages (add `.md` files to `docs/pages/`)

For customization (themes, authentication, custom pages), fetch `https://zuplo.com/docs/dev-portal/introduction`.

## Step 9: Deploy to the edge

### Option A: CLI deployment

```bash
# Login first (one-time)
zuplo login

# Initialize project on the platform
zuplo init

# Deploy
zuplo deploy
```

### Option B: Git integration (recommended for teams)

1. Connect your repo in the Zuplo Portal: **Settings > Source Control > Connect to GitHub**
2. Every push triggers a deployment
3. Each branch gets its own environment (e.g. `main` = production, feature branches = preview)

### CI/CD deployment

```bash
export ZUPLO_API_KEY=zpka_xxxxx
zuplo deploy --environment production
```

**Docs:** Fetch `https://zuplo.com/docs/articles/step-4-deploying-to-the-edge` for full deployment guide.
